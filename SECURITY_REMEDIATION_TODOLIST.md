# Komari Agent Security Remediation Todo List

Date: 2026-05-26

## Purpose

本清单基于 `SECURITY_AUDIT_REPORT.md` 的审计结论整理，目标是把风险项转化为可执行、可排期、可验收的整改任务。

适用范围：

- 控制面认证与传输安全
- 本地凭据持久化与安装方式
- 自动更新与发布供应链
- 远程命令执行、终端、ping 能力的边界收敛
- 日志、测试、文档与运维落地

建议执行原则：

- 先做止血类操作，再改协议和代码，再补测试与文档
- P0 项没有完成前，不建议继续扩大 agent 部署范围
- 每个阶段结束后都要做一次回归测试和灰度验证

## Phase 0: 立即止血与风险收敛

目标：先降低已知高风险面，避免在整改期间继续暴露控制面。

时间建议：0-3 天

### 0.1 凭据与部署盘点

- [ ] 盘点所有当前已部署 agent 的 token 来源：命令行、环境变量、`auto-discovery.json`、systemd/OpenRC/procd/launchd、Docker 启动参数。
- [ ] 盘点所有可能记录 `token=` URL 的系统：反向代理、CDN、WAF、日志平台、APM、HTTP 代理、WebSocket 代理、shell history、CI 日志。
- [ ] 识别是否存在开启 `--ignore-unsafe-cert` 的节点，并按环境列出主机清单。
- [ ] 识别是否存在开启 Web 终端/远程命令执行但业务上并不需要的节点。

### 0.2 立即运营措施

- [ ] 对已暴露在命令行、日志、服务配置、代理日志中的 token 执行轮换。
- [ ] 立即停止在生产部署示例中使用 `--token` 明文命令行传参。
- [ ] 立即停止在任何环境启用 `--ignore-unsafe-cert`，除非有书面例外审批。
- [ ] 对不需要远程终端的节点临时启用 `--disable-web-ssh`。
- [ ] 如自动更新尚无校验机制，临时关闭自动更新或改为人工审批更新。

### 0.3 现场加固

- [ ] 检查已部署主机上的 `auto-discovery.json` 是否对非 owner 可读；对现存文件立即收紧权限。
- [ ] 检查 systemd unit、OpenRC、launchd plist、Windows 服务参数中是否包含 token；如有，迁移到更安全的载入方式。
- [x] 清理 Docker 启动命令、运维手册、截图、排障文档中的明文 token 示例。
- [ ] 如代理或网关会记录完整 URL，先临时添加规则屏蔽 `token` 查询串。

### 0.4 阶段验收

- [ ] 已完成 token 轮换，旧 token 全部失效。
- [ ] 生产环境不存在 `--ignore-unsafe-cert` 实例。
- [ ] 非必要节点已关闭远程终端能力。
- [ ] 已识别并记录所有残留风险点，进入下一阶段整改。

## Phase 1: 控制面认证与传输重构

目标：消除 token 查询串传递，并建立更合理的认证与 TLS 约束。

时间建议：3-7 天

### 1.1 HTTP/WS 认证迁移

- [ ] 将以下客户端请求从 `?token=` 迁移为 `Authorization: Bearer <token>`：
- [ ] `server/websocket.go` 主 WebSocket 上报连接。
- [ ] `server/websocket.go` 终端 WebSocket 连接。
- [ ] `server/basicInfo.go` 基础信息上报。
- [ ] `server/task.go` 任务结果回传。
- [ ] 与服务端同步修改接口，确保 Header 认证与 WebSocket 握手兼容。
- [ ] 设计兼容期策略：新旧协议是否并行、并行多久、何时移除查询串。

### 1.2 TLS 风险收敛

- [ ] 重新设计 `--ignore-unsafe-cert` 的行为，至少满足以下之一：
- [ ] 默认移除此选项。
- [ ] 或改为仅允许自定义 CA / 指纹固定，而不是完全跳过校验。
- [ ] 或在启用该选项时，自动禁用远程控制与自动更新。
- [ ] 统一清点所有网络客户端是否共享同一套 TLS 安全策略，避免部分路径漏改。

### 1.3 凭据生命周期治理

- [ ] 为 token 增加轮换说明和失效流程。
- [ ] 确认服务端支持 token 主动吊销、过期、重新发放。
- [ ] 审查日志、错误信息、调试输出，避免再输出完整认证材料。

### 1.4 阶段验收

- [ ] 仓库代码中不再存在控制面 `?token=` 请求。
- [ ] 代码中不存在默认可触发 `InsecureSkipVerify: true` 的生产路径。
- [ ] Header 认证在 HTTP 与 WebSocket 链路均已通过联调。
- [ ] 已完成一次灰度发布，代理/网关日志中不再出现 token 查询串。

## Phase 2: 本地凭据存储与安装链路整改

目标：收紧本地敏感数据落盘方式，消除命令行与服务配置中的长期泄漏。

时间建议：5-10 天

### 2.1 自动发现凭据存储整改

- [x] 将 `auto-discovery.json` 从二进制同目录迁移到专用配置目录。
- [x] Linux/macOS 使用仅 owner 可读写的权限模式。
- [x] Windows 使用受限 ACL，仅服务账户或管理员可访问。
- [x] 在启动阶段增加权限自检：权限过宽时拒绝启动或打印高强度告警。
- [x] 明确配置文件 owner 校验逻辑，避免被低权限用户替换或投毒。

### 2.2 安装参数与服务定义整改

- [x] 安装脚本禁止把 token 原样输出到日志。
- [x] 安装脚本禁止把 token 直接写进 systemd/OpenRC/procd/launchd/Windows 服务命令行。
- [x] 优先改为环境变量文件、权限受控的 JSON 配置文件或系统凭据存储。
- [x] 审核 `install.sh` 与 `install.ps1` 的所有日志输出点，加入敏感参数脱敏。
- [x] 审核 Dockerfile 与文档示例，移除 `-t token` 这类明文示例。

### 2.3 运维迁移工作

- [ ] 为现有部署提供配置迁移脚本或迁移说明。
- [ ] 提供旧服务定义到新配置文件模式的无损迁移方案。
- [ ] 为 Windows、Linux、容器三类部署分别给出迁移步骤。

### 2.4 阶段验收

- [ ] 新部署方式不再要求 token 出现在命令行。
- [ ] 自动发现凭据文件权限已在各平台通过验证。
- [ ] 安装日志、服务定义、示例文档中不再出现明文 token。
- [ ] 已验证升级旧部署不会导致 agent 离线或配置丢失。

## Phase 3: 自动更新与供应链完整性补强

目标：让下载、发布、更新三条链路都有完整性校验，并默认 fail closed。

时间建议：1-2 周

### 3.1 发布流程整改

- [ ] 在 `.github/workflows/release.yml` 中为每个发布产物生成 SHA256 校验文件。
- [ ] 为发布产物增加签名文件或可验证的证明文件。
- [ ] 将二进制、校验文件、签名文件一并上传到 Release。
- [ ] 明确签名私钥或签名流程的保管方式与轮换责任。

### 3.2 更新逻辑整改

- [ ] 在 `update/update.go` 中显式启用 `go-github-selfupdate` 的校验器。
- [ ] 校验失败时必须终止更新，不允许回退为“仅凭 TLS 信任继续执行”。
- [ ] 为更新失败增加明确日志，但日志中不应泄漏敏感上下文。
- [ ] 检查更新链路是否受 `--ignore-unsafe-cert` 影响，必要时彻底解耦。

### 3.3 安装脚本供应链整改

- [ ] `install.sh` 下载 agent 二进制后先验 hash/签名，再替换本地文件。
- [ ] `install.ps1` 下载 agent 二进制与 `nssm.exe` 后先验 hash/签名，再执行或注册服务。
- [ ] 对 `--install-ghproxy` 的使用增加安全说明，并明确只允许可信代理。
- [ ] 为离线安装提供官方校验流程，避免用户绕过完整性验证。

### 3.4 阶段验收

- [ ] 最新 Release 已包含校验资产。
- [ ] 客户端更新逻辑在校验失败时能稳定拒绝安装。
- [ ] 安装脚本在篡改样本上验证为 fail closed。
- [ ] 已完成一次从发布到安装到更新的端到端完整性演练。

## Phase 4: 远程控制能力边界收敛

目标：降低 token 一旦泄漏后的破坏半径，并防止认证后的资源滥用。

时间建议：1-2 周

### 4.1 默认能力调整

- [ ] 将远程终端/远程命令执行改为显式 opt-in，而不是默认开启。
- [ ] 为不同能力拆分开关，例如：基础监控、任务执行、终端、ping 探测分离控制。
- [ ] 若服务端支持，考虑为 token 增加最小权限范围，而不是所有 token 默认具备全部控制能力。

### 4.2 任务执行安全边界

- [ ] 将 `cmd.Run()` 改为 `exec.CommandContext`，为任务增加超时与取消机制。
- [ ] 为 stdout/stderr 增加大小限制，避免 `bytes.Buffer` 无界增长。
- [ ] 评估改为流式上报任务输出，而不是先全部缓存到内存。
- [ ] 对单机并发任务数增加上限与排队策略。

### 4.3 终端与 ping 能力约束

- [ ] 对终端会话数量、空闲超时、最大会话时长增加限制。
- [ ] 对 ping 任务增加目标范围、协议、端口、频率与并发限制。
- [ ] 评估是否需要禁止对内网敏感网段的默认探测。
- [ ] 为异常频率的控制请求增加客户端侧限流或熔断。

### 4.4 日志脱敏与审计最小化

- [ ] 移除 `server/task.go` 中对完整命令明文的默认日志记录。
- [ ] 默认仅记录 task ID、开始/结束时间、退出码、输出大小。
- [ ] 如果确需命令审计，改为显式开关并配套脱敏规则。

### 4.5 阶段验收

- [ ] 默认部署不再自带远程 shell/RCE 能力。
- [ ] 恶意大输出、长时间阻塞、批量任务压测下，agent 不会 OOM 或无限制拉高资源占用。
- [ ] ping/终端/任务已各自具备明确并发和超时边界。
- [ ] 默认日志不再包含原始敏感命令文本。

## Phase 5: 测试、文档与发布准入

目标：把整改结果固化进测试、文档和 CI，避免安全回归。

时间建议：1 周

### 5.1 自动化测试补齐

- [ ] 为 Header 认证迁移增加单元测试和集成测试。
- [ ] 为自动发现文件权限增加平台相关测试。
- [ ] 为更新校验逻辑增加正向与篡改样本测试。
- [ ] 为 `--ignore-unsafe-cert` 的限制行为增加测试。
- [ ] 为安装脚本敏感参数脱敏增加脚本级测试。
- [ ] 为任务超时、输出大小限制、并发限制增加回归测试。

### 5.2 测试基线治理

- [ ] 重构 `server/task_test.go` 中依赖公网/ICMP 的不稳定测试。
- [ ] 将外网依赖改为可控 mock、本地测试夹具或可跳过的集成测试。
- [ ] 在 CI 中区分单元测试、安全回归测试、可选网络集成测试。

### 5.3 文档补齐

- [ ] 扩充 `readme.md`，加入安全部署与最小权限运行章节。
- [ ] 明确说明 token 不应出现在命令行、历史记录、服务定义、Docker 启动参数和日志中。
- [ ] 增加证书信任模型说明：自定义 CA、证书固定、禁止跳过校验。
- [ ] 增加自动更新完整性说明与离线校验方式。
- [ ] 增加 token 泄漏后的应急处置与轮换手册。

### 5.4 发布准入与回归门禁

- [ ] 在 CI 中增加安全检查门禁：禁止新代码再引入 `?token=` 模式。
- [ ] 在 CI 中增加门禁：禁止 `InsecureSkipVerify` 在默认生产路径出现。
- [ ] 在发布前执行一次安全回归 checklist。
- [ ] 将本清单的关键验收项转化为发布阻断条件。

### 5.5 阶段验收

- [ ] CI 已覆盖本次整改的关键安全边界。
- [ ] 文档足以支持新部署按安全方式落地。
- [ ] 发布流程对关键安全回归具备阻断能力。

## Recommended Execution Order

- [ ] 第 1 优先级：完成 Phase 0 与 Phase 1。
- [ ] 第 2 优先级：完成 Phase 2 与 Phase 3。
- [ ] 第 3 优先级：完成 Phase 4。
- [ ] 第 4 优先级：完成 Phase 5，并将其纳入长期工程流程。

## Suggested Owners

- [ ] Agent 开发：协议改造、代码修复、测试补齐。
- [ ] 平台/服务端开发：接口兼容、token 生命周期、权限模型。
- [ ] 运维/SRE：部署迁移、证书治理、日志与代理清理、灰度发布。
- [ ] 发布负责人：校验资产、签名流程、发布门禁。

## Definition Of Done

- [ ] 审计报告中的高优先级问题均已完成代码或流程整改。
- [ ] 新旧部署迁移方案已验证，不影响现网稳定性。
- [ ] 关键修复均有测试或发布门禁保护。
- [ ] README 与运维文档已同步更新。
- [ ] 完成一次独立复核，确认不存在明显回归。
